# MacBridge 产品需求文档（PRD）

**版本**：v0.1 (初稿)
**日期**：2026-05-07
**状态**：需求规划中

---

## 1. 产品定位

MacBridge 是一款运行在 macOS 上的轻量级菜单栏工具，帮助 **从 Windows 迁移到 Mac 的用户** 快速适应 Mac 的键盘习惯。核心做法：**自动识别当前键盘类型**（Win 布局 / Mac 布局 / 笔记本自带），并按需重新映射最关键的几组按键，消除肌肉记忆错位带来的日常摩擦。

### 与 Karabiner-Elements 的差异

| 维度 | Karabiner-Elements | MacBridge |
|------|-------------------|-----------|
| 配置复杂度 | 高（需手写/导入 JSON 规则） | 零配置，开箱即用 |
| 功能广度 | 全量键位自定义 | 只做"Win→Mac 迁移"一件事 |
| 目标用户 | 键盘极客、老 Mac 用户 | Mac 新手、双系统使用者 |
| 自动识别键盘 | 需手动配置 device rule | 插上即识别并切换 profile |

## 2. 目标用户画像

**Persona A：迁移新手**
- 工作换了 Mac，之前用了 10+ 年 Windows
- 不愿意折腾配置文件，只想"能用就行"
- 最痛：`Cmd+C` 总按成 `Ctrl+C`，拇指位 `Cmd` 和 `Alt` 反了

**Persona B：双平台使用者**
- 公司发 Mac，家里用 Win 台式机
- 每天在两台机器间切换
- 最痛：肌肉记忆被两种键位反复撕扯

**Persona C：外接键盘党**
- 用 HHKB / Keychron / IKBC 等 Windows 布局外接键盘
- Mac 自带键盘和外接键盘位置不同
- 最痛：拇指位置的 `Cmd` vs `Alt` 永远按错

| 痛点 | 频次 | MVP |

## 4. 功能需求
　
### 4.1 MVP（v0.1）核心功能

#### F1. 键盘自动识别
- **行为**：插入/切换键盘时自动识别键盘类型（Apple 官方 / Windows 布局 / 未知）
- **识别依据**：USB VendorID + ProductID + 键盘描述字符串
- **识别失败降级**：提供手动选择（Win 布局 / Mac 布局 / 不处理）
- **记忆**：用户手动选择过的键盘下次自动应用相同 profile

#### F2. 修饰键交换（Win 布局专用）
- **默认规则**：
  - 键盘上的 `Win` 键 → 系统识别为 `Cmd`（Mac 原本就是这样）
  - 键盘上的 `Alt` 键 → 系统识别为 `Option`（Mac 原本就是这样）
  - ✅ **关键**：交换物理位置，让最靠近空格的键始终是 `Cmd`
    - Win 布局上顺序是 `Ctrl | Win | Alt | Space`
    - Mac 期望的操作是拇指按 `Cmd`（最靠近空格）
    - 所以要把 `Alt` 改成 `Cmd`，把 `Win` 改成 `Option`
- **开关**：单独可开关，默认开启

#### F6. 菜单栏 UI
- **显示**：当前识别的键盘类型 + 激活的 profile 名称
- **操作**：
  - 切换 profile（Win Mode / Mac Mode / Off）
  - 开关各项映射（F2、F5、F8）
  - 打开设置
  - 退出
- **风格**：极简，符合 macOS 原生设计语言

#### F7. 启动项
- 支持开机自启（默认开启，可关闭）

#### F8. Ctrl 快捷键语义映射（Win→Mac）
- **目的**：让 Win 用户保留 `Ctrl+C/V/X/Z` 等肌肉记忆，不必强制学习 `Cmd` 的新手位
- **与 F2 的关系**：
  - F2 = "位置交换，学新手位"（保留 Mac 原生 Ctrl 快捷键）
  - F8 = "保留旧手位，改键的语义"（牺牲部分 Mac 原生 Ctrl 快捷键）
  - 二者互补，可独立开关，也可同时启用（双保险）
- **白名单映射**（12 组，覆盖 >90% 日常场景）：
  ```
  Ctrl + C/V/X/Z/A/F/S/O/N/P/T/W  →  Cmd + 同键
  Ctrl + Shift + T                →  Cmd + Shift + T  （恢复已关闭标签）
  Ctrl + Shift + Z                →  Cmd + Shift + Z  （重做）
  ```
- **明确不映射**（避免冲突）：
  - `Ctrl+H`（Mac 原生 = 隐藏应用）
  - `Ctrl+Y`（Mac 原生 = 粘贴已删除内容）
  - `Ctrl+A/E/K/F/B/N/P/D`（Mac 全局支持的 emacs 风格文本导航）
  - `Ctrl+Space`（输入法切换）
- **应用黑名单**（完全不映射）：
  - 终端类：`Terminal.app`、`iTerm2`、`Ghostty`、`Alacritty`、`Warp`、`Kitty`、`Hyper`、`Tabby`
  - 原因：保留 `Ctrl+C` 中断进程、`Ctrl+D` EOF 等关键功能
  - **技术挑战**：VSCode / JetBrains 等 IDE 内嵌终端面板需要精细识别（可先整 app 排除，v0.2 再优化到面板级）
- **覆盖远程场景**：因为 F8 在 `CGEventTap` 层面工作，比 `hidutil` 所在的 HID 层高一级，所以 **远程操控软件（UU 远程、VNC、Moonlight 等）注入的键盘事件也能被捕获和改写**。F2 做不到这一点
- **开关**：单独可开关，**默认开启**（目标用户是 Windows 迁移者，Ctrl+C=复制 是最大肌肉记忆痛点；远程控制用户完全依赖它；终端黑名单已保护主要冲突场景）

### 4.2 v0.2 规划

- **自定义快捷键映射引擎**（取代目前 F8 硬编码白名单）：用户可增删改任意 `trigger → output` 规则，支持 per-app scope
- **Home/End 语义切换**（原 F5）：`Home` → `Cmd+←`，`End` → `Cmd+→`。默认关闭（Mac 原生语义是跳文档首尾，有争议）。等自定义映射引擎上线后作为一条内置可选预设实现
- 输入法切换键自定义（Shift / Ctrl+Shift / Caps）
- 截图快捷键映射（Win 的 PrintScreen → Mac 的 Cmd+Shift+4）
- "渐进迁移"模式：记录用户按错次数，提示"你已经 30 天没按错 Ctrl+C 了，可以关闭这条映射"
- 多键盘 profile 记忆

### 4.3 明确不做

- ❌ 键位自定义编辑器
- ❌ 宏录制
- ❌ 应用级规则（在不同 app 下不同规则）
- ❌ 云端同步
- ❌ Windows 版本（反向迁移是另一个产品）

## 5. 非功能需求

### 5.1 性能
- 键盘事件延迟 < 5ms（用户无感知）
- 常驻内存 < 30MB
- CPU 占用 < 1%（空闲时）

### 5.2 兼容性
- macOS 13 Ventura 及以上
- 同时支持 Intel / Apple Silicon
- 支持外接有线/蓝牙键盘

### 5.3 权限
- 需要 **辅助功能 (Accessibility)** 权限（键盘事件监听/重映射）
- 需要 **输入监控 (Input Monitoring)** 权限
- 首次启动引导用户授权，文案清晰说明为什么需要

### 5.4 稳定性
- 系统休眠/唤醒后映射不失效
- 键盘热插拔不需要重启应用
- 应用崩溃时键盘事件不被吞掉（降级到系统默认）

## 6. 技术方向（初步）

- **语言**：Swift + SwiftUI（菜单栏 app）
- **键盘事件层**：
  - 方案 A：`CGEventTap`（事件级拦截，轻量）
  - 方案 B：`IOHIDManager`（设备级，更底层，识别键盘更准）
  - 倾向：A + B 结合，B 做识别、A 做改键
- **分发**：DMG 直装 + 后续考虑 Homebrew Cask；Mac App Store 因沙盒限制基本不可行
